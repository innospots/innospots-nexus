# Module Skills

Module skills documentation is a developer-triggered snapshot of a module's
public capabilities. It is generated by scanning the selected module or
project directory as a whole; it is not maintained incrementally alongside
ordinary code changes.

## Generation Policy

- Do not create, edit, or synchronize `SKILL.md` or files under `references/`
  while implementing features, fixing bugs, refactoring, or changing packages.
- Adding, deleting, renaming, or moving Java types must not automatically
  trigger skills documentation changes.
- Generate or refresh skills documentation only when a developer explicitly
  requests a scan and identifies the module or project scope.
- A requested scan must inspect the complete selected source directory and
  regenerate the corresponding `SKILL.md` and `references/` set consistently.
- Do not infer a skills documentation request from a code task, documentation
  task, build failure, or stale skills content.
- Existing skills documentation may remain temporarily out of date between
  developer-requested scans.

When a scan is explicitly requested, each selected module uses a `skills/`
directory under `src/main/resources/skills/` containing a `SKILL.md` and a
`references/` subdirectory with per-package reference documentation.

## SKILL.md Format

Every `SKILL.md` must have YAML front matter and the following sections:

```yaml
---
name: <module-artifact-id>
description: |
  Concise summary followed by a bullet list of every major capability with
  enough detail to understand what the module can do.
version: <semver>
---
```

### Required Sections (in order)

| Section | Content |
|---------|---------|
| `# <Module Name>` | Module title |
| `## Module Overview` | Brief introduction + "What it can do" capability table |
| `## Class Reference` | Per-package tables: `\| Class \| Type \| Description \|` |
| `## References` | Mapping table to per-package reference files in `references/` |

The capability table in Module Overview should have two columns:
`| Capability | Description |` — one row per cohesive feature.

### Example Capability Table

```markdown
## Module Overview

**What it can do:**

| Capability | Description |
|------------|-------------|
| **Condition Engine** | Build filter/rule conditions with typed factors and operators |
| **Identity & Access** | Model users, roles, groups, and organizations |
| ... | ... |
```

### Example Class Reference

```markdown
### Package `domain.condition`

| Class | Type | Description |
|-------|------|-------------|
| `Factor` | `class` | A single filter criterion with placeholder resolution |
| `Operator` | `enum` | Comparison operators with dual DB/script symbols |
```

## References Directory

Each package in the module gets a corresponding markdown file in
`references/` named `<package-name>.md` (replace dots with hyphens,
e.g. `domain-condition.md` for `domain.condition`). The file documents
every public class/enum/interface/record/annotation in the package:

```markdown
# Package Name

## ClassName

**Type:** class/enum/interface/record/annotation

Description of the class purpose.

### Method/Field Name
- **Signature:** `methodName(ParamType param) → ReturnType`
- **Description:** what this method does
- **Parameters:** (if applicable) param name + type + description
- **Returns:** description of return value
```

## Consistency Rules

- Consistency is required within one explicitly requested scan result, not
  after every source-code modification.
- A full module scan should document every public API package in scope.
- A generated `SKILL.md` class reference and its `references/` files must
  describe the same scanned source snapshot.
- Do not manually patch individual skills entries to follow isolated code
  changes; rerun the developer-requested directory scan instead.
